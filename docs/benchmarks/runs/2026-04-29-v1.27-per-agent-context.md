### L2.1 — cross-file middleware (clean)

| Run | Plan | Files | Validation | Commit | Wall | Score |
|---|---|---|---|---|---|---|
| #1 | 1 step | `middleware/request-log.ts`, `server.ts` | pass | yes (`e056fe1`) | 6m | 10/10 |
| #2 | 1 step | — (nothing landed) | pass¹ | **none** | 10m | **1/10** |

¹ Validation "passed" trivially because nothing was modified.

**#2 — catastrophic regression.** Coder emitted `{action: "modify", path: "src/middleware/request-log.ts", edits: [...]}` for a file that does not exist on disk. SafeWriter throws (`Cannot modify non-existent file`), retry-with-real-content correctly skips (no on-disk content to feed Fixer), `unrecoveredWrites` collects the path, and nothing else writes either. The task reports `done` (rightly so — partial completion is now signalled by v1.28's `commit_partial`, which did fire). But the working tree on the auto-branch is empty.

In the v1.26 prompt — with newlySources in Coder's view but the same `coderContext` shape — the model never made this mistake. Two runs in a row Coder emitted `action: "create"` correctly. Once the Architect's design field became impoverished (lean context), the Architect's design no longer carried the cue "this file does not exist yet → create, not modify". Coder relied on Architect to disambiguate file state and got the wrong steer.

**Variance band:** v1.26 L2.1 = `[10, 10]` mean 10 zero variance. v1.27 lean L2.1 = `[10, 1]` mean 5.5 spread 9.

## Decision: revert lean context, keep Planner few-shot

The data is decisive after 5 lean-context runs (2 × L1.1, 2 × L2.1, 1 × L2.3):
- Two atomic L1.1 runs both 3-5× slower than v1.26 baseline
- L2.1 went from `[10, 10]` zero-variance to `[10, 1]` wide spread; the 1 was a structural failure (modify-on-non-existent-file) directly traceable to Architect context starvation
- L2.3 cumulative produced a malformed file the Fixer couldn't recover

Hypothesis killed: **Architect's "design" field is load-bearing for Coder.** When the Architect's input is rich enough that it can name specific files and patterns in its design, Coder follows. When the Architect's input is just conventions + repo-map, the design becomes generic ("create middleware, register it"), Coder has to fill in details from its own context, and small drift compounds across the prompt.

The Reviewer and Tester probably could have tolerated lean context independently, but the test was bundled with Architect changes and the regression didn't separate cleanly.

**Reverted in [orchestrator.ts](../../../packages/agents/src/orchestrator.ts)** — Architect/Reviewer/Tester returned to v1.26 contexts (full ragSnippets + newlySources). Planner few-shot in [planner.ts](../../../packages/agents/src/planner.ts) **retained** — it's purely additive prompt content, no quality regression observed.

## What worked

- **Decisive empirical falsification of the lean-context hypothesis.** Five runs made the data unambiguous: lean Architect/Reviewer prompts produce slower, lower-quality output. We now know not to chase that path.
- **Planner few-shot retained without observable regression.** L2.3 sanity post-revert showed Planner picking 1 coupled step (matching Example A's pattern). On a clean main and on cumulative state, runs match v1.26 baseline. Few-shot for Planner is a permanent addition with zero downside.
- **v1.28's `commit_partial` event proved its worth here.** L2.1 #2 lean run landed nothing on disk; previously this would have been silent. The `commit_partial` event with `unrecoveredWrites: ["src/middleware/request-log.ts"]` made the failure obvious from the SSE stream.

## What broke

- **Lean Architect context degraded Coder output.** Coder's quality is downstream of Architect's design field; impoverishing Architect's view starved Coder of specificity (file existence, exact patterns). This is a clear architectural lesson: per-agent context tailoring isn't free, the agents share information through their handoffs.
- **L2.3 #1 lean run produced a malformed routes/users.ts.** Three patches accumulated incorrectly; Fixer rounds didn't unwind. v1.25.1's try/catch correctly prevented a hard task fail (commit_skipped instead). But the working-tree state is unusable until the auto-branch is reset.
- **All 5 lean-variant runs were 2-3× slower than v1.26 baseline** at the same model and same prompts otherwise. Less context produced longer / wandering generations, not shorter ones — counter-intuitive but reproducible.

## Lessons / next iteration ideas

1. **Per-agent context tailoring is more delicate than it looks.** The agents form a pipeline: Architect → Coder → Reviewer → Fixer. Each agent's output becomes the next agent's input. Stripping context from Architect doesn't just save tokens for Architect; it impoverishes the design that Coder consumes. Token savings need to be measured against pipeline-level quality, not per-agent.
2. **If we revisit context tailoring later**, do it one agent at a time with a benchmark gate. Reviewer alone (without Architect changes) is the safest first candidate — it sees the full patch as a parameter, so its context can plausibly be lean. Architect and Tester need their context preserved.
3. **Planner few-shot is a no-brainer keep.** It joins the Coder/Fixer few-shot from v1.26 as the "show, don't tell" pattern that consistently works on local models.
4. **The malformed routes/users.ts in L2.3 #1 is interesting on its own.** Even if lean context was the trigger, the patch-application accumulating into a duplicated/nested mess hints at a possible safety improvement: after each `applyEdits` call, run a quick syntactic sanity check (e.g. brace balance) before declaring success. Out of scope for v1.27 — file as a future hardening idea.

## Decision

Land v1.27 in tree as **Planner-few-shot only**. Document the lean-context revert as a recorded experiment with empirical data. Bench scores match v1.26 baseline; we kept a clean improvement (Planner) and shed a clear regression (lean Architect/Reviewer/Tester).

Phase 2 closes here. Phase 3 architectural work (tool-calling Coder, sub-agents, iterative editing) is justified by what we now know:
- Prompt-engineering reaches a plateau (we've gone L2.1 to deterministic 10/10 with few-shot, can't do better at this layer).
- Cross-step coordination on cumulative state (L2.3) still varies between 7.9 and 9.0 means with the right plan vs the wrong plan; stronger architecture (sub-agents per task type, tool-calling Coder) is the next lever.
- Variance from Reviewer rejections also surfaced in v1.26 — also points to architectural rather than prompt fixes.

Run a v1.21–v1.27 regression sweep (full benchmark suite, all five iterations) before starting Phase 3, to baseline the cumulative gains before architectural changes start moving things again.
