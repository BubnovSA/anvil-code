# Run 2026-04-29 — v1.30.5 Verify-syntax tool after replace_in_file

Targeted at the structural-placement failure surfaced in v1.30.4 (model
consumed a closing `});` when picking line range, leaving the file unbalanced).
After every `replace_in_file` (and at `create_file` time), the dispatcher
runs a brace/paren/bracket balance check on the affected file. If a previously-
balanced file becomes unbalanced post-edit, the WorkingSet rolls back and the
tool result tells the model what went wrong, so the model can retry with
different coordinates within its same loop — before validation/Fixer ever fires.

## What v1.30.5 changed

[packages/agents/src/tool-calling-coder.ts](../../../packages/agents/src/tool-calling-coder.ts):

- **`checkBraceBalance(content)`** — returns `{ok: true, balance}` or `{ok: false, reason, balance}`. Walks the source character-by-character, tracking string/line-comment/block-comment/template-literal state so braces inside `"strings"`, `// line comments`, `/* block comments */` are ignored. Counts `{}/()/​[]` triplets; early-exits on net-negative (closing without opener mid-file). Returns informative reason like `"unclosed '{': net 1"` or `"extra closing ')'"`.
- **`WorkingSet.overwriteRaw(path, content)`** — preserves the existing entry's action label (modify/create) but resets content. Used for undo.
- **`replace_in_file` dispatcher path** — captures pre-edit content + balance check on entry. After applying the replace, re-checks balance. If file was balanced before AND is unbalanced after, calls `overwriteRaw(path, before)` to restore + returns error message naming the issue. Model retries within its loop.
- **`create_file` dispatcher path** — pre-checks the supplied content. Refuses to create with reason if unbalanced.
- Skips check for non-source files (markdown, json, yaml, etc.) — only `.ts/.tsx/.js/.jsx/.mjs/.cjs`.

14 new unit tests across `checkBraceBalance` (8 cases) and dispatcher integration (6 cases): balanced/unbalanced detection, string/comment awareness, the v1.30.4 "consumed closing brace" pattern specifically, rollback behavior, balanced-edits-unchanged-pass-through, non-source-file skip, create_file pre-check.


The brace count is balanced — every `{` has a matching `}`. The check correctly says "ok" because the count is right. But the file is structurally a mess: duplicate `/health` handler header, orphaned body lines, /version inserted in the middle. Net braces match → balance check passes → no rollback fires.

**What this is.** A different class of bug from v1.30.4. v1.30.4: edit consumed a brace (net imbalance, caught by v1.30.5). Now: edit DUPLICATES content into new_text (model thought it should "preserve surrounding context" by including the /health line in its new_text, then the new_text replaced lines that were AFTER /health, leaving the original /health intact + a copy in the new_text). Balance is preserved — duplicate text balances itself.

This isn't fixable with brace-balance alone. The lighter signal is "new_text shouldn't repeat verbatim lines from the surrounding context that aren't in the replaced range." That's a content-similarity check, not a syntax check.

## What worked

- **First ever completed task on rag-system-target.** v1.30.5 + v1.30.3.1 stack kept Ollama up through 32 min. We've now graduated from "task_failed (infra)" to "commit_skipped (output quality)" — a much better failure class because it's recoverable and informative.
- **Brace balance check correctly catches the v1.30.4 failure mode.** Verified by direct unit test against the actual broken-file pattern from that run-file.
- **String/comment awareness works.** Tests cover braces inside strings, line comments, block comments, escape sequences.
- **Rollback is clean.** Pre-edit content is restored on rejected edits; the model sees a fresh state on retry, never sees the broken intermediate.
- **No regression on baseline.** Balanced edits pass through unchanged; non-source files (md, json) skipped.
- 294/294 unit-tests green.

## What broke

- **Duplicate-content failure mode.** v1.30.5 doesn't address it because braces remain balanced when the model's new_text duplicates lines from the surrounding context. The file is structurally broken in a way brace balance can't see.
- **Fixer still struggles** with structurally-broken-but-balanced files. ~30 min of retries didn't recover this one. The duplicate pattern isn't pinpointed by typecheck/test errors, so Fixer iterates without convergence.
- **Wall time grew** to 32 minutes. Verify-syntax adds zero overhead per call but the model still does its 25-call budget when convergence is hard.

## Lessons / next iteration ideas

1. **Brace balance is necessary but not sufficient.** It catches the count-imbalance class. It can't catch the duplicate-content class. We need a separate signal for "your replace_in_file new_text contains lines that already exist in the surrounding context" — a content-similarity check on the replace boundary.
2. **v1.30.6 candidate — duplicate-content detection** (~2-3 hours):
   - After replace_in_file applies, scan the 5-10 lines immediately before `start_line` and after `end_line` (in the post-edit content). If `new_text` contains any of these as exact-line matches, refuse with `"new_text duplicates context lines (line K already exists at the boundary). replace_in_file should NOT include the surrounding context — it replaces the named line range only."`
   - This explicitly captures the "I'll preserve the surrounding code by including it in new_text" misconception.
3. **Or move up the abstraction ladder.** We've spent 6 sub-iterations whack-a-moling Coder failure modes. Each one is real and the fixes are sound, but the cycle suggests the underlying problem is "Coder reasoning about line coordinates is fragile." A bigger architectural move (v1.31 — split `read_file` into `read_function`/`read_class` with structural anchors instead of line numbers) might break the cycle by removing line coordinates entirely.
4. **First milestone is real:** task `completed` instead of `task_failed`. This means the operator can now inspect the auto-branch and see what the model attempted, even though the result isn't merge-quality. UX improvement worth noting.

## Decision

**v1.30.5 lands in tree.** The architectural change (verify-syntax + rollback) is correct, unit-tested against the specific v1.30.4 failure pattern, and produced a measurable infra-stability improvement (first completed task on rag-system-target, no Ollama crash).

**Recommend a strategic pause** on the v1.30.x prompt-and-tooling micro-fixes after this. Six iterations have peeled six failure modes; the seventh (duplicate content) suggests we're in a long tail of Coder failure modes that each take a half-day to identify and fix. Two productive paths forward:

- **v1.30.6 — duplicate-content detection** (continue the whack-a-mole, ~3h). Cheap, addresses this specific failure.
- **v1.31 — sub-agents and/or structural anchors** (architectural move, days). Replace line-coordinate edits with symbol-anchored edits (`replace_function('UserService.list', new_body)`) — eliminates whole classes of placement bugs.

The choice is "keep iterating on the line-coord primitive" vs "move past line coords entirely." The benchmark cost of each path is similar. The architectural path produces a cleaner end state.
