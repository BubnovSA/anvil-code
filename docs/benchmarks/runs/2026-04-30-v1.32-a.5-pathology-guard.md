# Run 2026-04-30 — v1.32-a.5 Coder/Fixer pathology guard

Direct response to v1.32-a.4 robustness bench, where run #1 took **58 minutes**
because Coder spent the bulk of that time spinning on near-identical
`replace_in_file` calls that brace-balance kept rejecting. The same pattern
v1.32-a.1 surfaced (44-min outlier on its first L4.1 attempt). The fix:
detect "stuck on same (tool + path) tuple with repeated errors" and break
early via a strategy nudge, then a hard bail after 2 such cycles. Symmetric
implementation in Coder and Fixer.

**Result on a 3-run L4.1 sample:** the guard fires correctly on a stuck
Fixer (bounded wall to 23 min vs the 58-min v1.32-a.4 outlier), and is
inert on a healthy run (clean 6-min byte-perfect commit, identical to
v1.32-a.4 run #4). One run hit an unrelated Ollama infra crash
("fetch failed") not attributable to v1.32-a.5.

## What changed

### Detection logic

Both Coder.execute() and Fixer.execute() track three new state variables in their tool-call loops:

```ts
let consecutiveSameToolErrors = 0;
let lastErrorFingerprint = '';
let pathologyStrikes = 0;
```

After every dispatched tool call, if the result starts with `error:`:
- Compute fingerprint = `${call.function.name}:${path-or-file-arg}` (loose enough to catch "stuck on same file" even when the model varies start_line / end_line / new_text between retries — exactly the L4.1 brace-balance retry pattern)
- If fingerprint matches `lastErrorFingerprint`, increment counter; otherwise reset to 1 with the new fingerprint
- When counter reaches `PATHOLOGY_THRESHOLD` (5):
  - Increment `pathologyStrikes`
  - Reset counter and fingerprint
  - If `pathologyStrikes >= MAX_PATHOLOGY_STRIKES` (2) → log warning, set `pathologyBail = true`, break inner loop
  - Otherwise → push a strategy-change nudge into messages, allowing the loop to continue

A non-error tool call (success or `done()`) resets both counter and fingerprint to zero.

The strategy nudge:
> You have called {tool} on "{path}" 5 times and gotten the same kind of error each time. The current approach is not working — change strategy. Try one of: (a) different start_line/end_line on a re-read of the file (line numbers shift after every edit); (b) a different tool — structural ones (add_route / add_method / add_import) often handle cases where replace_in_file struggles; (c) read_file the path to verify the current state; (d) call done() if you cannot make further progress. Do not retry the same call shape again.

The Fixer variant of the nudge replaces "add_route / add_method / add_import" with "add_method / replace_method / add_import" (more relevant to fixing).

### Why the chosen thresholds

- **`PATHOLOGY_THRESHOLD = 5`**: low enough to bound wall time aggressively, high enough that healthy iteration isn't interrupted. The L4.1 bench shows Coder/Fixer typically need 2-4 calls per file before getting it right; 5 gives margin without disguising real pathology.
- **`MAX_PATHOLOGY_STRIKES = 2`**: one nudge gives the model a clean chance to switch strategy; if it still spins, the pathology is real and continued retries are pointless. Total budget = 10 same-fingerprint errors before bail.

Both constants are exported from `tool-calling-coder.ts`. Tests reference them directly so they double as documentation; future tuning is a one-line change.

### Tests (+5 new, 392 total)

**Coder — 4 tests:**
- `pushes a strategy nudge after PATHOLOGY_THRESHOLD consecutive same-fingerprint errors` — fake router emits the same `replace_in_file` call repeatedly; verifies nudge appears in messages on the (THRESHOLD + 1)th invocation
- `hard-bails after MAX_PATHOLOGY_STRIKES same-fingerprint cycles` — same fake router; verifies the loop terminates after exactly THRESHOLD × MAX_STRIKES (10) invocations
- `a successful tool call resets the consecutive-error counter` — alternating pattern (4 errors + 1 success per cycle); verifies pathology never fires; loop runs full MAX_TOOL_CALLS budget
- `a different-fingerprint error resets the counter` — alternates between two failing paths; verifies pathology never fires

**Fixer — 1 test:**
- `hard-bails Fixer loop after MAX_PATHOLOGY_STRIKES same-fingerprint cycles` — symmetric Fixer test verifying the imported constants drive identical behavior

### Run #1 — pathology guard validates

Coder phase produced a cosmetic edit on routes/users.ts (variable rename + reply.code(201).send pattern). Reviewer approved. TestRunner failed on the assertion `expect(user.createdAt).toBeTruthy()`. Fixer attempt #1 ran ~15 min trying various edits to user-service.ts — alternating `replace_function` (rejected because `create` is an object-literal method, not a top-level FunctionDeclaration) and `replace_in_file` (with progressively worse `new_text` shapes, including unbalanced extra `}`). Fixer attempt #2 entered pathology — same fingerprint `replace_in_file:src/services/user-service.ts` repeated for 5+ consecutive errors, nudge fired, model didn't take it, hit 2 strikes, **bailed at 14 tool calls**. Validation correctly flagged still-failing → commit_skipped.

**Without the guard**, this run would have continued retrying until either Fixer's MAX_TOOL_CALLS=25 budget ran out, or Ollama crashed from prolonged pressure. The 58-min v1.32-a.4 run #1 was exactly this pattern. v1.32-a.5 cuts that to ~23 min — wall-time bounded by design.

`{"status":"failed","result":"fetch failed"}` partway through the run. The llama runner subprocess died (the v1.30.3 ceiling). Independent of v1.32-a.5; happens occasionally on long tool-call sessions. Counted as an infra failure, not a v1.32-a.5 regression.

## What worked

- ✅ **Guard fires correctly when pathology is real.** Run #1 had Fixer in a clear retry loop on `replace_in_file:src/services/user-service.ts` — guard bailed at 14 calls (2 strikes × 5 + reset overhead). Wall bounded to 23 min.
- ✅ **Guard is inert on healthy runs.** Run #3 had no pathology activity, completed in 6 min with byte-perfect diff identical to v1.32-a.4 run #4. The guard adds zero overhead when the model behaves.
- ✅ **Strategy nudge gives the model a clean chance to recover** before bail. The nudge is concrete (suggests structural tool / re-read / done) rather than abstract. The model didn't take it on run #1, but if it had — bail would have been avoided.
- ✅ **Symmetric Coder + Fixer implementation.** Both agents share the same constants (`PATHOLOGY_THRESHOLD`, `MAX_PATHOLOGY_STRIKES`) and the same logic shape. Future tuning affects both consistently.
- ✅ **+5 unit tests** with broad coverage: nudge fires, bail fires, success resets counter, different-fingerprint resets counter, Fixer mirrors Coder.

## What broke

- ⚠ **Run #1 commit_skipped because the guard cut Fixer off mid-pathology.** This is the *correct* outcome (continued spinning would have just delayed the same outcome by 30+ min), but it does mean the v1.32-a.5 commit-rate measure looks worse than v1.32-a.4's 5/5. **What the rate actually means:** v1.32-a.4 was fortunate that the model eventually escaped its retry loops on every run (typically just before MAX_TOOL_CALLS); v1.32-a.5 catches the spin and bails earlier. Both produce the same commit-rate over a large enough sample if the model's recovery rate is constant; v1.32-a.5 just costs less wall time when it bails.
- ⚠ **Run #2 Ollama infra crash** is independent of v1.32-a.5 but worth noting — long Fixer sessions can still trigger the v1.30.3 llama-runner OOM. Now that pathology bails earlier, the runner is under less prolonged pressure → infra crashes should be *less* likely under v1.32-a.5 than v1.32-a.4. Unverified with this small sample.
- ⚠ **Pathology fires only on consecutive same-fingerprint errors.** A model that alternates between two failing fingerprints (e.g. replace_function ↔ replace_in_file on the same path) wouldn't trigger the consecutive counter. The L4.1 v1.32-a.5 run #1 trace shows some such alternation early on, before the model settled into pure replace_in_file pathology that triggered the guard. Possible future tightening: track recent N calls and fire on >K% same fingerprint, not strictly consecutive. **Tracked as a future micro-improvement, not a v1.32-a.5 blocker.**

## Lessons / next iteration ideas

1. **Bounded failure is better than unbounded retry.** The L4.1 v1.32-a.4 robustness bench had a real wall-time tail (58 min on run #1) that came from the model retrying doomed edits. v1.32-a.5 turns that into a faster, cleaner failure. The total throughput on long bench cycles improves accordingly.
2. **Pathology is a model-side problem, not a tool-side problem.** The guard catches it; it doesn't fix it. The model's reasoning gets stuck because the prompt doesn't guide recovery from "edit got rejected" well. Two tractable directions:
   - **Better feedback in the rejection message.** When brace-balance rolls back an edit, the message could include the specific imbalance ("net +1 curly after your edit") AND a hint ("did you include the wrapping function header in new_text?"). Currently it just says "structurally unbalanced".
   - **Grammar-constrained tool calls in a future backend swap** (llama.cpp direct) would prevent the model from emitting unbalanced new_text in the first place — JSON Schema can constrain new_text to a valid syntactic substring. Phase 4 candidate.
3. **Threshold tuning may be worth revisiting after more data.** Current `THRESHOLD = 5, STRIKES = 2` was chosen by judgment, not measurement. A larger sample (10+ L4.1 runs) might suggest tighter or looser values. Cheap to tune; defer until we have more evidence.
4. **Guard composition with the no-tool-calls retry (v1.32-a.3) is clean.** Both work via `consecutiveX` counters and progressively-strong-nudge-then-bail. They cover orthogonal failure modes (text-only vs error-loop) without interference. Future sub-iterations on the same loop can follow the same pattern.
5. **The L4.1 robustness vs latency tradeoff** is now explicit: v1.32-a.4 favored eventually-completing-even-if-slow; v1.32-a.5 favors fail-fast. For an interactive operator, fail-fast wins. For an unattended batch processor, the older behavior might be preferable. Worth keeping `MAX_PATHOLOGY_STRIKES` configurable via env in the future if the use case shifts.

## Decision

**v1.32-a.5 lands.** The guard delivers its promise: pathology is bounded, healthy runs are unaffected, the failure signal arrives faster. Trade-off accepted: a small fraction of runs that *would have* eventually fixed the bug after 50+ tool calls now bail at 10 calls. The wall-time savings on the dominant failure mode outweigh that loss.

**Recommended next step:** the v1.32-a.x sub-iteration arc should pause here. The Phase 3 closure path is now visible:

- **v1.32-a.6** (~1h, opt) — `prettier --write` post-step in orchestrator. Cosmetic-only, would convert the v1.32-a.4 minor-style runs into byte-perfect outcomes. Cheap quick-win.
- **v1.32-c** (~1 week) — sub-agents (BugFixAgent / RefactorAgent / FeatureAgent) on v1.31 structural primitives. The Phase 3 architectural closure. Bigger investment with multi-task generalization upside.
- **Phase 4** — storage upgrades (Qdrant, SQLite symbol table) + backend swap to llama.cpp direct with grammar-constrained tool calls. Performance + reliability frontier.

The natural priority is **v1.32-c** next — Phase 3 architectural closure makes the system feature-complete on its current foundation. Phase 4 then becomes about making the foundation faster and more reliable, not adding new architectural primitives.
