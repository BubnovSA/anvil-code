## Architectural changes built during the run

### v1.21 — Context fidelity
- `glob` ignored node_modules → `cwd` + relative pattern fix (was indexing 2490 files; now 5)
- `TypeChecker`/`TestRunner` now use `writer.root` (was `config.projectRoot` — wrong target in multi-project)
- `COMMIT_ONLY_IF_VALID=true` — main never gets a broken commit
- Graceful Tester (`try/catch`) + `TESTER_ENABLED` flag — one bad LLM JSON no longer kills a step
- `ProjectConventions` reader (`testFramework`, `moduleType`, `tsStrict`, `needsJsSuffix`, `entryPoints`)
- `buildPromptContext` — Project Conventions → Existing Files (full source) → RAG snippets → Design
- `===== BEGIN FILE =====` markers (uppercase, not valid TS — model can't accidentally copy them)

### v1.22 — Cross-step consistency
- `executeStep` accepts `previousChanges` snapshot from earlier steps
- "Recently modified by previous steps" block in prompt — supersedes "Existing project files"
- Planner rule: same-file steps MUST be sequential (deps required), cross-file coupled = single step
- AST parser now indexes `export const X = {...}` (object literals, factories, classes) — was missing
- `PLANNER_MAX_STEPS` cap — emergency brake on over-planning

### v1.23 — Patch-based editing (the big one)
- `FileChange` is now a discriminated union: `create | modify | delete`
  - `create`: full content (as before)
  - `modify`: list of `{search, replace}` edit blocks — applied in order
  - `delete`: just a path
- `applyEdits()` core: search must match exactly once (zero or multiple matches abort)
- `SafeWriter.execute` switches on `action`; `modify` reads disk + applies edits + writes
- Per-path dedupe before write — multiple changes on same file merge into atomic apply
- Retry-with-real-content: when applyEdits fails, Fixer is called once with the literal current file content as context
- Entry-point files always included in `ragFilePaths` — fixes Coder hallucinating server.ts content
- Coder/Fixer prompts rewritten with explicit edit-block format + 14 rules
- Tester schema restricted to `action: 'create'` (it never modifies existing source)
- 196 unit tests passing after migration; 10 new tests for `applyEdits` edge cases

## Key findings

1. **Model size dominates prompt engineering.** 14+ rules in the Coder prompt didn't pull deepseek-coder-v2:16b through L2.1; gemma2:27b cleared it on the first try. qwen2.5-coder:32b is the first model to land L2 cross-file with a green commit on the new patch-based format.

2. **Coder-spec ≠ uniformly better.** qwen2.5-coder:32b beat gemma2:27b on L2.1 (clean) but didn't beat it on cumulative — both regressed, with different failure modes.

3. **Patch-based editing is the one-way safety win.** Across all three models, every "search not found" attempt left the file untouched. Compare to v1.22: deepseek-coder-v2:16b destroyed server.ts five times in a row (rewrote it as `export default app` with `require()` in ESM, lost `app.listen()`, wrote `// Existing code...` placeholders).

4. **Cumulative state regresses every model.** Adding even one prior commit to the sandbox shifts model output enough to break tasks that were green on a clean main. Suspected cause: more context surface → more hallucination targets. Not solvable by another rule.

5. **Validation catches typecheck and tests, not runtime.** A server.ts variant that compiled but never called `listen()` passed validation in one run. The pipeline needs a runtime smoke step (out of scope for this run).

6. **New failure mode at the patch-based level: search minification.** qwen2.5-coder:32b on L2.3 cumulative wrote multi-line existing code as a single line in `search` blocks, which strict-match rejects every time. Then the retry Fixer emitted empty `search` strings (Zod-rejected). The model is "compressing" search to save tokens.

7. **Silent partial completes.** L2.2 on qwen-coder cumulative: schema created → committed, integration into routes failed retry → file unchanged → task reported `done`. The user has no visible signal that half the work didn't land. Needs a `commit_partial` event.

## What works (production-ready)

- L1 atomic on clean sandbox, qwen-coder:32b — stable across 3 task types
- L2 cross-file on clean sandbox — green for the first time (L2.1)
- L3 single-file refactoring — gemma2:27b 10/10 baseline
- COMMIT_ONLY_IF_VALID + patch-based — main is structurally protected, no broken-code commits

## What is fragile

- L2.2 refactor (extract + integrate) — silent partial failure pattern
- L2.3 multi-file on cumulative state — search minification kills it on qwen-coder, hallucinated property accesses on gemma2
- TesterAgent on `vitest` projects — keeps emitting `as jest.Mock` despite explicit guidance; currently disabled by default

## Lessons / next iteration ideas

- Repo-map in every prompt (Aider pattern) — top hypothesis for breaking the cumulative ceiling
- Whitespace-tolerant matching as a fallback in `applyEdits` — directly addresses the minification failure mode
- Few-shot examples in Coder/Fixer prompts (instead of more rules) — small models follow examples better than abstract instructions
- Per-agent context tailoring — Reviewer should see a diff, not full files; saves tokens and is more focused
- `commit_partial` SSE event when retry-with-feedback fails on at least one file
- Tool-calling Coder (read_file / replace_in_file / run_test) — long-term architectural shift; physically prevents most failure classes
