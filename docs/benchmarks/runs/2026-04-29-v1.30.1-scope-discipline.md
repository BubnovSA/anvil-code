Hardening pass on v1.30. v1.30 broke the scale ceiling (target file edits land
correctly even on a 91-file project) but introduced a new failure mode:
**scope creep**. On the rag-system /version task v1.30 wiped `package.json`

`__tests__/` directory with random content. The model treats the project as
fair game once the patch-format constraint is gone.

`create_file` / `delete_file` to a path not in scope (or to a hardcoded
forbidden list of config files).

## What v1.30.1 changed

### Path extraction
[packages/agents/src/tool-calling-coder.ts](../../../packages/agents/src/tool-calling-coder.ts):

```typescript
export function extractAllowedPaths(taskDescription: string): Set<string>;
```

Regex picks anything that looks like a relative path with a known source-file extension. Trailing punctuation, surrounding backticks/quotes/parens are stripped. Leading `./` is normalized away. **Critical regex bug fixed during implementation:** alternation `js|json` is left-to-right, so `.json` was matching `.js` (the prefix) and dropping `on` â€” added `\b` after the extension to require a full word match. 4 unit tests cover paths with directories, multi-file tasks, surrounded-by-punctuation paths, no-paths-mentioned (returns empty), and the package.json corner case.

### Forbidden list
```typescript
const ALWAYS_FORBIDDEN_PATTERNS = [
  /(?:^|\/)package\.json$/,
  /(?:^|\/)package-lock\.json$/,
  /(?:^|\/)pnpm-lock\.yaml$/, /(?:^|\/)yarn\.lock$/,
  /(?:^|\/)tsconfig.*\.json$/,
  /(?:^|\/)vitest\.config\.(?:ts|js|mjs|cjs)$/,
  /(?:^|\/)jest\.config\.(?:ts|js|mjs|cjs)$/,
  /(?:^|\/)\.env(?:\..+)?$/,
  /(?:^|\/)turbo\.json$/, /(?:^|\/)\.gitignore$/,
];
```

A forbidden path can still be touched if the operator names it explicitly in the task ("update packages/api/package.json to add zod" â†’ `package.json` is in `policy.allowed` and the forbidden check is bypassed).

### Dispatcher enforcement
`dispatchToolCall(call, ws, policy)` â€” `read_file` is unrestricted; `replace_in_file` / `create_file` / `delete_file` go through `isWriteAllowed(path, policy)`. On rejection the tool result is `error: path "X" is not named in the task description â€” only [Y, Z] are in scope`, which the model sees as a tool message and adapts.

### System prompt rewrite (took two passes)

First pass had strong language: "your tool calls will be REJECTED" + "DO NOT retry... Stop, finalize the changes you've made (or call done() with no changes), and let the operator adjust." Result on the live benchmark: model produced **0 file changes** on /version and getSize (deterministic across two runs). The escape hatch "or call done() with no changes" gave the model permission to bail.

Second pass softened the language and removed the bail-out: "If the dispatcher rejects a write, focus your work on a path that IS in scope. ... You MUST complete the substantive change requested in the task. Calling done() without making any of the requested edits is wrong unless the task is genuinely a no-op." After this fix the model resumed making edits.

## What worked

- **Scope discipline solves the v1.30 collateral-damage failure mode.** Tool-calling Coder now writes ONLY to paths the task names. Verified by direct comparison: same task as v1.30, same model, only delta is the policy.
- **Path extraction is robust.** 6 unit tests across naming styles (with/without quotes, parens, leading `./`, mid-prose mentions) all pass; regex `\b` anchor prevents the `.json` â†’ `.js` truncation that bit the first implementation.
- **Forbidden list with operator opt-in is the right API.** Default-deny on package.json/tsconfig/lockfiles, but explicit task mention bypasses. Lets the system be safe by default and permissive when the operator means it.
- **268/268 tests green.** v1.30 baseline + 14 new for v1.30.1 path/policy logic.

## What broke

- **System prompt language matters for inaction risk.** First pass with strong "REJECTED" + "give up if blocked" â†’ model bailed entirely (0 file changes, deterministic). Removing the bail-out fixed it. Lesson: scope-policy prompts must have a clear positive instruction ("focus on allowed paths") and not a negative escape hatch.
- **Coder cargo-culting.** /version returned `{status, ollama, uptime}` (the /health body) instead of `{ version: '1.0.0' }`. Model read /health for context (correct workflow), then copied its handler instead of writing the new logic. This is a task-understanding bug, not scope. Possible fix: stronger Coder prompt example showing "read for context, don't copy as template."
- **Patch-based validation Fixer is unrestricted at scale.** When the tool-calling Coder produced a (questionable) edit to server.ts, the patch-based validation Fixer kicked in to "fix" the validation failures and started emitting search/replace patches against unrelated test files (`from 'jest'` imports that don't exist anywhere) â€” exact same failure mode as v1.29 patch-based Coder. None landed (search-not-found), but the policy/system prompt of tool-calling does NOT extend to the patch-based Fixer used in `runValidationLoop`. v1.30.x â€” Fixer migration to tool-calling â€” is now urgent.
- **Reviewer still approves empty Coder output.** First v1.30.1 run had Coder produce 0 file changes; Reviewer approved them. v1.30.2 target.
- **Wall times.** /version on the 91-file target is now 5-14 minutes. Tool-calling loop with multiple LLM round-trips per task is the dominant cost.

## Lessons / next iteration ideas

1. **The full v1.30 storyline is now coherent:**
   - v1.29 proved patch-based scales badly.
   - v1.30 proved tool-calling fixes the patch ceiling, at the cost of scope creep.
   - v1.30.1 fixes scope creep, surfacing the next layer of issues (model comprehension + Fixer not migrated).
2. **Migrate Fixer to tool-calling (v1.30.x â€” high priority).** The validation-loop Fixer is the same patch-based agent we replaced in Coder. It hallucinates `search` blocks at scale exactly like Coder used to. Should reuse `ToolCallingCoderAgent` infrastructure with a Fixer-flavored system prompt + same WorkingSet / dispatch / policy machinery.
3. **Reviewer empty-output rejection (v1.30.2 â€” small fix).** Cheap prompt edit: if `currentChanges.length === 0`, return `isApproved: false` with issue "Coder produced no file changes." Independent of all the above.
4. **Cargo-culting fix.** Coder system prompt could include: "read_file is for understanding context; the new code's content comes from the task description, not from sibling routes. If the task says return X, return X â€” don't copy the shape of an adjacent handler."
5. **Validation infrastructure on rag-system-target.** Tests/typecheck currently fail on the snapshot for reasons unrelated to the model output (missing tooling, environment differences). For v1.30+ benchmarks, the target needs `npm install` + a working test/typecheck baseline, otherwise validation always fires and Fixer always runs (and pollutes the working tree).

## Decision

**v1.30.1 lands in tree.** The empirical claim â€” "scope discipline prevents collateral writes" â€” is decisively confirmed by the matched-pair comparison with v1.30 on the same task. The tool-calling Coder is now safe at scale: it writes only what the operator asked for.

**Next iteration: v1.30.2 (Reviewer rejects empty output, ~1h) â†’ v1.30.3 (tool-calling Fixer, ~3-5h).** After Fixer migration, re-run the rag-system benchmark suite â€” expect the patch-based-Fixer-hallucinated-jest-imports failure mode to disappear, and we'll get a cleaner read on what's left.

**Default still off:** `TOOL_CALLING_CODER=false` by default. Sandbox-scale users keep deterministic patch-based path. The scale benefits (and v1.30.1 safety) are opt-in per-task or per-deployment.
