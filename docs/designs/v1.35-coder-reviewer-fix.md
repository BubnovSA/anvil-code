# Design — v1.35 Coder/Reviewer fix (multi-file unblock)

> **Status:** DRAFT — pre-impl. **Date:** 2026-05-09.
> **Order:** после v1.34 hybrid-search. Цель — разблокировать L2 (multi-file), который v1.34 не сдвинул.
> **Links:** диагноз — [docs/benchmarks/runs/2026-05-08-l2x-multifile.md](../benchmarks/runs/2026-05-08-l2x-multifile.md).

## TL;DR

L2.x bench 2026-05-08 дал 0/8 commits и 1/8 корректного кода. Pipeline-диагностика показала: Committer **работает корректно** (его блокирует `COMMIT_ONLY_IF_VALID=true` после провала `tsc --noEmit`), а слабое звено — Reviewer (LLM-judge без AST-парса) и Coder (генерит синтаксически битый TS, либо 0 файлов). Вставляем `tsc --noEmit` **внутрь Reviewer-loop'а до LLM-judge'а** + fail-fast на 0 файлов + информативные fail-сообщения. Цель: 4+/8 корректных коммитов на повторном L2.x.

## 1. Goals & non-goals

| | |
|---|---|
| **G1** | Pre-Reviewer TS-check ловит parse-errors до LLM-judge'а; Fixer получает строку+сообщение от tsc, а не "approved" |
| **G2** | Coder с writtenFiles.length===0 после step → step помечается failed (не quiet completed) |
| **G3** | task `result` при failed содержит конкретный last review.issues / validation.errors, а не `"All N steps failed; aborting"` |
| **G4** | L2.x bench rerun: ≥4/8 валидных коммитов (Сценарий A) |
| **NG1** | Не трогаем Planner / retrieval / BM25 — диагноз показал, что они не виноваты |
| **NG2** | Не трогаем commit-gate (`COMMIT_ONLY_IF_VALID`) — он работает по дизайну |
| **NG3** | Не запускаем L3.x (5+ файлов) до закрытия G4 |
| **NG4** | Не меняем Reviewer-промпт — добавляем pre-check вокруг него, не вместо |

## 2. Архитектура

**Текущий поток (orchestrator.ts:225-262, reviewer.ts:12-38):**
```
per step:
  Coder → write files → Reviewer-LLM (3x retry с Fixer) → emit step_done
после всех шагов:
  runValidationLoop() → tsc --noEmit + tests (2x retry) → commit gate
```

**Новый поток:**
```
per step:
  Coder → write files
  if writtenFiles.length === 0: throw "step produced no changes" (G2)
  Pre-Reviewer: tsc --noEmit on changed files only (G1)
    if parse-error: feed errors → Fixer → re-run; до 2 попыток
  Reviewer-LLM (3x retry с Fixer)
после всех шагов:
  runValidationLoop() остаётся как safety net
commit gate без изменений
fail-message обогащается last issues (G3)
```

- [packages/safe-exec/src/type-checker.ts](../../packages/safe-exec/src/type-checker.ts) — добавить partial-check API (только изменённые файлы)

**Findings (code-reading, 2026-05-09):**
> **L2.7 (email filter) — код ок, commit нет:** `runValidationLoop` запускает `tsc --noEmit -p tsconfig.json` на полном проекте (orchestrator.ts:725-752). Sandbox компилируется чисто на main (exit 0 подтверждено). Но генерированный код использует `request.query.email` без typed Fastify-schema → strict-mode TS2339/TS2345. Validator fails → `shouldSkipCommit=true` (orchestrator.ts:227) → commit не создаётся. Гипотеза 3 подтверждена: проблема в tsc-fail, не в commit-механизме.
> **L2.4/L2.6 (no-op):** Coder (tool-calling path) возвращает `ws.toFileChanges()` = `[]` — WorkingSet не имеет изменений (LLM только `read_file` + `done()`). `codeChanges.files.length === 0` → `currentChanges = []`. Reviewer получает пустой массив файлов → `filesSummary = ""` → LLM approves empty review. Step complete, 0 file changes. Outer step-5 write: ничего не записывает. Commit: `writtenFiles.length > 0` = false → no commit. Status=completed. Гипотеза 2 подтверждена.
> **L2.2/L2.5 (false-positive):** Reviewer — чистый LLM-judge (reviewer.ts:12-38), видит FileChange как текст. Не запускает tsc. Синтаксически битый код (метод врезан в середину другого) может выглядеть логично в диффе → Reviewer approves. Validation fails. `shouldSkipCommit=true`. Гипотеза 1 подтверждена.

## 4. Acceptance criteria

- [ ] **AC1** — Pre-Reviewer TS-check ловит parse-errors L2.5-class на unit-тесте; tsc-error пробрасывается в Fixer.
- [ ] **AC2** — Coder с 0 файлов даёт `failed` (не `completed`); SSE stream имеет `step_noop` event.
- [ ] **AC3** — Failed task `result` содержит конкретные issues (regex test: `result.includes("Step ") && result.length > 50`).
- [ ] **AC4** — L2.x rerun: ≥4/8 коммитов в auto-веток с корректным TS, который компилируется.
- [ ] **AC5** — Wall-time медианной задачи не вырос >20% относительно baseline (pre-check секунды на 2-3 файла).

## 5. Risks & mitigations

| Risk | Severity | Mitigation |
|---|---|---|
| `tsc --noEmit` на изменённых файлах падает из-за зависимостей с другими файлами проекта | MED | Запускать с `--project` указывая sandbox tsconfig, не отдельные файлы; tsc разрешит импорты сам |
| Pre-check добавит секунды × 3 retry на каждый step → wall-time +10-30% | MED | Запускать после первого write; кэшировать tsc через incremental build (`--incremental`) |
| Fixer не справится с tsc-errors (промпт-формат не подходит) | MED | B3 — отдельный test cycle, в случае провала — fallback к LLM-judge'у с tsc-output как контекст |
| Pre-check ловит false-positive (валидный код, но tsc warning) | LOW | Различать `error` vs `warning` в TscResult; reject только на errors |

## 7. Alternatives considered

- **Заменить LLM-judge Reviewer на pure tsc + tests.** Отвергнуто: LLM-judge ловит семантические проблемы (плохая архитектура, нарушение контракта), которые AST не видит. Берём оба.
- **Добавить tsc в Coder loop, не в Reviewer.** Отвергнуто: Coder уже большой prompt; чище отделить генерацию от валидации.
- **Использовать TS Compiler API программно вместо spawn'а tsc.** Отвергнуто на эту итерацию: больше кода, +зависимость; spawn простой и работает в текущем `safe-exec`.
